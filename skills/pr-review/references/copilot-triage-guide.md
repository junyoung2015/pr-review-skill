# GitHub Copilot Comment Triage Guide

How to parse, validate, and classify GitHub Copilot PR review comments.

---

## Data Sources

Prefer these sources in order:

1. `*.review-data.json` sidecar generated by `scripts/fetch-pr.sh`
2. Direct GitHub API responses (`gh api repos/<repo>/pulls/<pr>/reviews` and `.../comments`)
3. Markdown copied into `docs/pr-for-review/*.md`

For Copilot, structured data matters. Markdown alone is usually not enough because the summary review and inline comments are exposed separately in GitHub.

---

## How GitHub Copilot Usually Appears

Current GitHub PR review payloads commonly expose Copilot as:

- **Summary review** author/login: `copilot-pull-request-reviewer[bot]`
- **Inline comment** author/login: `Copilot`

Do **not** rely on author strings alone. Always join inline comments to their review using `pull_request_review_id`.

---

## Parsing GitHub Copilot Sections

### 1. Summary Review

From reviews API:

```json
{
  "id": 3934347775,
  "user": { "login": "copilot-pull-request-reviewer[bot]" },
  "body": "## Pull request overview ...",
  "submitted_at": "2026-03-12T06:49:08Z"
}
```

Use this for:

- overall context
- file-by-file summary
- total comment count
- identifying the latest Copilot review round

### 2. Inline Review Comments

From pull request comments API:

```json
{
  "id": 2922644132,
  "user": { "login": "Copilot" },
  "pull_request_review_id": 3934347775,
  "path": "apps/web-v2/src/views/login/lib/load-script.ts",
  "line": 10,
  "body": "loadScript() resolves immediately ..."
}
```

Use these for actual triage. The summary review is not actionable by itself unless it references a concrete inline comment.

---

## Latest Round Selection

Copilot may review multiple rounds of the same PR.

Always:

1. Sort Copilot reviews by `submitted_at`
2. Pick the latest review as the active round
3. Keep only comments whose `pull_request_review_id` matches that latest review

Dismiss older Copilot comments with:

`이전 리뷰 라운드 기준이라 현재 검토 대상 아님`

This avoids duplicate triage when Copilot is re-run after new pushes.

## Validation Process

For each Copilot comment:

### Step 1: Check if the issue still exists

Read the actual file and the current diff.

Common stale cases:

- The author already fixed it in a later commit
- Copilot commented on an earlier version of the patch
- The latest review round superseded an earlier one

### Step 2: Check scope

If the file is OUT_OF_SCOPE for the target ticket:

- dismiss with `스코프 외 ([TICKET] 커밋 소속)`
- exception: still flag obvious crash/security/data-loss issues

### Step 3: Check conventions and review instructions

Cross-reference:

- `CLAUDE.md`
- `.github/copilot-instructions.md`
- `.github/instructions/**/*.instructions.md`

Dismiss when Copilot is pushing against an intentional project rule or a path-specific instruction.

### Step 4: Check if the suggestion is correct

Copilot comments are often useful, but they can still be wrong:

- type/schema mismatch comment is valid only if the generated type truly lacks the field
- missing env var comment is valid only if the variable is still consumed in runtime code
- concurrency/race comment is valid only if existing guards do not already serialize access

Never accept a Copilot suggestion without reading the current code.

## Classification Decision Tree

```text
Is this comment from the latest Copilot review round?
├── No → DISMISSED
│   Reason: "이전 리뷰 라운드 기준이라 현재 검토 대상 아님"
└── Yes
    ├── Does the issue still exist at HEAD?
    │   ├── No → DISMISSED
    │   │   Reason: "이후 커밋에서 이미 수정됨"
    │   └── Yes
    │       ├── Is it out of scope?
    │       │   └── Yes → DISMISSED
    │       │       Reason: "스코프 외 ([TICKET] 커밋 소속)"
    │       ├── Does it conflict with repo convention or Copilot instructions?
    │       │   └── Yes → DISMISSED
    │       │       Reason: "프로젝트 컨벤션과 상충"
    │       └── Is the fix mechanical?
    │           ├── Yes → FIX-SELF
    │           └── No
    │               ├── Would the developer learn something important?
    │               │   ├── Yes → PASS-TO-CREATOR
    │               │   └── No → FIX-SELF
```

Then assign severity independently:

- 🔴 HIGH: crash, broken flow, incorrect behavior, type break that blocks build/runtime
- 🟡 MEDIUM: missing guards, incomplete contract update, weak error handling
- 🟢 LOW: docs drift, missing env docs, cleanup, minor test hygiene

## Zero Copilot Comments Protocol

When the selected review source includes Copilot but the latest Copilot review has 0 inline comments:

1. State it clearly in the review:

   ```markdown
   > GitHub Copilot 코멘트가 없습니다 (미실행, 자동 리뷰 미설정, 또는 최신 라운드 코멘트 없음).
   ```

2. Do **not** treat this as a blocker
3. Continue with the independent deep code review as the primary source of truth

Possible reasons:

- Copilot review was not requested
- automatic review is not enabled for the repo
- Copilot only produced a summary comment
- the latest round reviewed the PR but found no inline issues

---

## Recommended Output Fields

For each Copilot triage row, capture:

- `provider`: `copilot`
- `review_id`
- `comment_id`
- `path`
- `line` or line range
- `summary`: one-line Korean summary of Copilot's concern
- `decision`: `fix-self`, `pass-to-creator`, or `dismissed`
- `reason`
- `severity`

This keeps downstream reply/resolve scripts provider-neutral.
